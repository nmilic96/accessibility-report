⚠
The site does not conform to WCAG 2.1 Level AA. Zero pages passed automated testing across 105 scanned URLs. However, the raw error count is significantly inflated by three site-wide structural issues — duplicate SVG IDs in the navigation, the accessibility widget's own unlabelled buttons, and low-contrast text in the header — that affect every single page. Fixing those three issues would eliminate the majority of the 1,675 recorded violations. The remaining issues are mostly contained, targeted fixes rather than widespread problems.
Scope note: This is an automated audit covering the Croatian-language pages (/hr). Automated tools detect approximately 30–40% of WCAG issues. Issues not detectable by automation include: keyboard focus order and trapping, screen reader announcement quality, meaningful alternative text quality, cognitive load, and reflow at 320px. A complete conformance assessment requires manual testing. PDF documents linked from the site were not tested.
Key Findings
🔴 3 Issues Affect Every Page
Colour contrast failures in the page header, duplicate SVG IDs in the language switcher flags, and unlabelled buttons in the accessibility widget (#a11y-panel) are present on all 105 scanned pages. Fixing these three root causes removes the bulk of the violation count.
⚠ Accessibility Widget Fails Its Own Test
The "Alati za pristupačnosti" overlay panel — intended to improve accessibility — itself generates WCAG 4.1.2 violations because its line-height toggle buttons have no accessible names. This is a known problem with third-party accessibility overlays and is ironic given their purpose.
📋 Forms Are Keyboard-Inaccessible
Multiple filter forms (news, procurement, employment, EU projects) use JavaScript onchange auto-submit without a submit button. Keyboard-only users cannot submit these forms, violating WCAG 3.2.2. This affects core site functionality.
🎬 Video Has No Captions
The NAPCORE EU project page embeds a video (/videos/napcore.mp4) without captions or a transcript. This is a WCAG 1.2.2 critical violation. All prerecorded video content must have synchronised captions.

Critical Video element has no captions 1.2.2 1
page ▼ EXPAND
Prerecorded video content must have synchronised captions so that deaf and hard-of-hearing users can access the audio content.
Affected element
<video src="/videos/napcore.mp4" controls></video>
— on /hr/eu-projekti/instrumenti-za-povezivanje-europe/6-napcore-move-b4-...
How to fix
Add a <track kind="captions" src="napcore.vtt" srclang="hr"> element inside the <video> tag with a WebVTT caption file. If captions cannot be produced quickly, add a text transcript on the same page as an interim measure. This is a Level A requirement.
Serious Insufficient colour contrast 1.4.3 105
pages ▼ EXPAND
Text elements must have a contrast ratio of at least 4.5:1 against their background (3:1 for large text ≥18pt or ≥14pt bold). Multiple elements fail this throughout the site.
Examples (all pages)
  • Page header: H1 and H2 text over a background image — white text but insufficient due to semi-transparent overlay
  • Breadcrumb navigation links (e.g. "Početna stranica", "EU projekti") in the page header region
  • Cookie notice: body text, "Saznaj više" link, and "Shvaćam" button
  • News card dates and category tags (e.g. "Oglasi", "17/04/2026")
  • Homepage category headings ("EU projekti", "Plan nabave", etc.) over background
  • Contact page: form dropdown placeholder text (1:1 ratio — invisible)
  • Google Maps attribution text and controls on the contact page
How to fix
Page header: Darken the overlay behind header text, or darken the text itself. The main H1/H2 text appears to be white on a light-ish image background without sufficient contrast. Use a colour contrast checker (e.g. WebAIM Contrast Checker) to verify 4.5:1.

Cookie banner: The cookie notice uses a light grey text on a near-white background. Darken the text to at least #595959 on white.

Contact form dropdown: The custom select "-- Odaberite temu upita --" has a 1:1 contrast ratio (white text on white background). This is completely invisible without CSS.
Serious Duplicate id attribute values 4.1.1 105
pages ▼ EXPAND
ID attributes must be unique within a page. Duplicate IDs break ARIA relationships, break label associations, and confuse assistive technologies that use IDs to navigate the DOM.
Examples
  • id="flag-icons-hr" — the Croatian flag SVG is inlined twice in the page (navigation renders the same SVG for the desktop and mobile versions)
  • id="flag-icons-gb" — same issue with the UK flag for language switching
  • id="Group", id="Vector", id="Vector_2", id="Vector_3" — SVG icon elements inlined multiple times with the same IDs
  • id="nav_element_139" — procurement navigation card duplicated on some pages
How to fix
For SVG flags/icons: use the <symbol> + <use> pattern — define the SVG once with unique IDs, then reference it with <use href="#flag-icons-hr">. This avoids duplication entirely.

For the mobile/desktop navigation duplication: render only one nav and use CSS to show/hide it, rather than rendering two separate navs with the same content and IDs.
Serious Buttons with no accessible name 4.1.2 105
pages ▼ EXPAND
Button elements must have an accessible name so screen readers can announce their purpose. These are the accessibility widget's own controls — a particularly notable failure.
Affected elements (in #a11y-panel)
<button class="a11y-widget__option js-a11y-line is-active" data-line="normal" aria-pressed="true">
<span class="a11y-icon">...</span> <!-- no text, no aria-label -->
</button>

Same pattern for data-line="large" and data-line="xlarge"
How to fix
Add aria-label to each button:
• data-line="normal" → aria-label="Normal line height"
• data-line="large" → aria-label="Large line height"
• data-line="xlarge" → aria-label="Extra large line height"

Similarly, the font size buttons (A, A, A) should have descriptive aria-labels.

Add text-decoration: underline to the "Saznaj više" link in the cookie notice. Alternatively, add a visible border-bottom. This ensures users who cannot perceive colour differences (colour-blind users) can still identify the link.
Serious Forms without submit button 3.2.2 7
pages ▼ EXPAND
Forms that auto-submit on input change (without a submit button) prevent keyboard-only users from submitting them. Changing a select element value with arrow keys triggers immediate form submission, which is a context change the user did not initiate.
Affected pages
  • /hr/novosti and news filter pages — year/category selects use onchange="runFireSubmit(this)"
  • /hr/stranice/javna-nabava/arhiva — procurement archive filters
  • /hr/stranice/javna-nabava/javne-nabave — procurement list filters
  • /hr/stranice/o-nama/zaposljavanja — employment search
How to fix
Add a visible <button type="submit">Pretraži</button> to each filter form. Remove the onchange auto-submit behaviour. The submit button gives keyboard users a predictable point of control and avoids unexpected context changes.
Serious Form fields missing labels 1.3.1 / 4.1.2 8
pages ▼ EXPAND
Every form input must have a programmatically associated label. Placeholder text disappears on input and is not a substitute for a label element.
Examples
  • EU projects search: <input placeholder="Pretražite sve projekte" name="search_projects"> — no label
  • Procurement search: <input placeholder="Pretraži po nazivu" id="search_term"> — no label
  • Spending info page: hidden input id="payee-name-disabled" — no label
  • Employment search input — no label
How to fix
Add <label for="search_projects">Pretraži projekte</label> before each input, or add aria-label="Pretraži projekte" directly on the input element. If the label should be visually hidden, use a CSS visually-hidden class rather than removing it entirely.

Serious Invalid list structure 1.3.1 1
page ▼ EXPAND
List elements (<ul>, <ol>) must only directly contain <li> elements. A nested <ul> without a parent <li> is invalid HTML and breaks screen reader list navigation.
Affected page

How to fix
Wrap the inner <ul> in a <li> element: <ul><li><ul><li>...</li></ul></li></ul>. This is likely a content authoring error in the CMS rich text editor.

Moderate Deprecated align="" attribute 1.3.1 5
pages ▼ EXPAND
The align="" attribute is obsolete in HTML5 and conveys presentational information in markup rather than CSS. This indicates content pasted directly from Word documents into the CMS rich text editor.
Affected pages
  • /hr/stranice/zahtjevi-i-suglasnosti/izvanredni-prijevoz/4 — fee table with align="center" on all cells
  • /hr/projekti/1-multimodalna-platforma-... — align="justify" on all paragraphs
  • /hr/eu-projekti/projekti/55, /7-napcore-x, /43 — align="justify" paragraphs
How to fix
Configure the CMS rich text editor (likely a WYSIWYG like TinyMCE or similar) to strip the align attribute on paste and replace it with CSS text-align. For existing content, a one-time find-replace on the database or a CSS rule to override the attribute is the quickest fix: p[align], td[align] { text-align: revert; }
Serious reCAPTCHA textarea has no accessible name 4.1.2 1
page ▼ EXPAND
A hidden textarea generated by reCAPTCHA (id="g-recaptcha-response") has no associated label. While hidden visually, it is not hidden from assistive technologies.
Affected element
/hr/stranice/kontakt

<textarea id="g-recaptcha-response" name="g-recaptcha-response"
style="display:none"></textarea>
How to fix
Add aria-hidden="true" to the reCAPTCHA textarea since it is hidden and not intended for user interaction. This removes it from the accessibility tree. Alternatively, add aria-label="reCAPTCHA response" if it must remain in the tree.
